# 《梦境支配者》P0 核心数值系统合理性评估与优化建议

> 基于《梦境支配者》P0 核心数值系统详细设计 v1.0 评估。本文档只做评估与优化建议，不替代详细数值文档本身。

## 1. 评估结论

当前核心数值系统的方向是合理的，已经具备 P0 可验证的数值骨架：主人睡眠度承担唯一失败条件，猫咪血量承担单猫生存压力，队伍屎意值承担节奏打断，队伍饱肚度承担补给循环。四项数值分别对应「守床」「换猫」「猫砂盆」「喂食器」四类玩家行动，能把单张守床地图从纯清怪玩法拉成持续决策循环。

但当前版本存在一个主要风险：压力源之间叠加过密，且多个惩罚都指向行动能力下降。如果睡眠度下降、猫咪低血量、屎意高、饱肚低同时发生，玩家会进入「越慢越处理不了、越处理不了越慢」的负反馈。P0 阶段建议保留四项数值的结构，但先降低惩罚耦合强度，把调试重点放在节奏窗口是否舒服，而不是一次性把所有边界规则做满。

总体建议是：核心结构保留，惩罚强度下调，调试变量收敛，先用 5 到 7 个关键旋钮验证 P0 手感。

## 2. 合理性判断

| 模块 | 当前判断 | 合理点 | 主要隐患 |
|-|-|-|-|
| 主人睡眠度 | 保留 | 唯一失败条件清晰，能稳定驱动守床目标 | 上限惩罚偏重，容易让一次失误滚成整局崩盘 |
| 猫咪血量 | 保留并简化 | 单猫独立血量能支持切猫与站位决策 | 虚弱叠加、恢复、无敌帧规则偏多，P0 实现和调试成本较高 |
| 队伍屎意值 | 保留 | 是项目差异化记忆点，能制造强制脱战窗口 | 高阶段减速、攻速、冷却一起降，容易让玩家失去补救能力 |
| 队伍饱肚度 | 保留但弱化惩罚 | 能把喂食器和技能释放纳入资源循环 | 与屎意值共同叠惩罚时，会把战斗节奏压得过慢 |
| 10 层曲线 | 保留框架，重做调参目标 | 结构上符合 P0 路线节点玩法 | 第 7 到第 9 层提升较陡，叠加本能数值后可能过压 |

这套系统最大的优点是「每个数值都能对应一个明确操作」。玩家看到睡眠度低，会守床和清怪；看到血量低，会切猫或治疗；看到屎意高，会找猫砂盆；看到饱肚低，会找喂食器。这个对应关系很重要，因为 P0 不能依赖大量关卡内容撑深度，必须让基础系统自己产生变化。

问题在于，当前每项数值都设计了较完整的惩罚链：睡眠度有归零失败和上限惩罚，血量有虚弱不可上场，屎意有移动 / 攻速 / 冷却下降和拉屎惩罚，饱肚有伤害下降和技能压力。单看都合理，叠加起来会偏重。P0 应先保证玩家能理解、能补救、能形成节奏，再逐步加深惩罚。

## 3. 关键数值风险

### 3.1 睡眠度上限惩罚偏硬

当前「拉屎一次，睡眠度上限永久 -20」的惩罚辨识度很强，但它对新手和 P0 调试都偏硬。一次失误从 100 降到 80 还可接受，两次就只剩 60，后续 Boss 战如果按文档中「Boss 战睡眠度可能压到 5 到 15」的目标推进，玩家会很快进入几乎不可逆状态。

建议 P0 第一版不要直接用固定 -20。更稳的做法是把第一次惩罚调成 -15，第二次及以后再使用 -20；或者保留 -20，但把最低上限从 20 临时提高到 40，等确认整体难度后再下调。这样既保留「不能不管屎意」的严肃性，也不会让一次早期失误直接破坏整局体验。

### 3.2 屎意高阶段惩罚过多

屎意值在 86 到 100 时同时给 -30% 移速、-20% 攻速、-20% 技能冷却惩罚。这个组合有表达力，但玩家真正需要补救时，最需要的是跑向猫砂盆、清掉拦路怪或释放救场技能。此时同时削弱移动、输出和技能，会让玩家补救能力被一起打掉。

建议 P0 中只保留一个主要惩罚维度：屎意高主要影响移动和少量攻击节奏，不再额外压技能冷却。技能冷却惩罚可以留到后续版本，或只在拉屎倒计时最后 5 秒出现。这样能保留「憋不住」的体感，又不至于让玩家完全失去处理手段。

### 3.3 饱肚度与屎意值的负反馈容易叠爆

饱肚度低会降低战斗能力，喂食器会恢复饱肚度，但同时触发消化并加速屎意增长。这条循环是好的，因为它把「吃」和「拉」连接成项目特色。但如果饱肚度低造成输出下降，而屎意高又造成移动和攻击下降，两个系统同时告急时，玩家可能既清不动怪，也跑不到猫砂盆。

建议 P0 把饱肚度的惩罚定位改成「输出效率下降」，不要再影响移动或技能冷却。屎意值负责移动压力，饱肚度负责输出压力，两者分工清晰，叠加时也更容易读懂。

### 3.4 虚弱规则过完整，P0 可能不需要全部实现

当前虚弱状态包括 25 秒不可上场、恢复 25% HP、三猫全虚弱保底、同猫连续虚弱叠加、虚弱期间治疗无效、自动切换和无敌帧。这套规则从完整游戏角度是合理的，但 P0 第一版实现成本偏高，也会增加 QA 组合。

建议 P0 第一版先简化为：猫血量归零后虚弱 20 秒；虚弱结束恢复 30% HP；虚弱期间不可上场；当前猫虚弱时自动切到血量最高的可用猫；切换获得 0.3 秒无敌。连续虚弱叠加、三猫全虚弱特殊缩短、治疗无效细则可以先不做或只作为后续增强。

### 3.5 第 7 到第 9 层难度斜率需要实测重校

文档中第 7 层难度系数为 1.35，第 8 层为 1.50，第 9 层为 1.65，同时屎意 / 饱肚速率也提高到 1.10 到 1.15。这会让战斗压力和本能压力同步上升。同步上升适合后期挑战，但 P0 的问题是玩家还在学习系统，过早同步加压容易掩盖到底是哪条数值出了问题。

建议第一轮灰盒测试中让怪物强度和本能压力分开调。第 7 到第 9 层可以先只提高怪物数量和血量，暂时不提高屎意 / 饱肚速率；等确认常规战斗节奏后，再打开本能速率倍率。

## 4. 推荐的 P0 调整方案

### 4.1 先收敛到 7 个核心调试旋钮

| 调试旋钮 | 当前文档值 | P0 建议 | 目的 |
|-|-|-|-|
| 睡眠度小怪接触伤害 | 3 | 保留 3，先只调怪物数量 | 先验证守床压力，不让伤害和数量同时变动 |
| 睡眠度拉屎上限惩罚 | -20 | 首次 -15，后续 -20；或最低上限先设 40 | 减少早期一次失误毁局 |
| 屎意自然增长率 | 0.4 / 秒 | 前 3 层 0.3 / 秒，第 4 层后 0.4 / 秒 | 给教学期更多理解时间 |
| 屎意消化倍率 | ×2.0 | 保留 ×2.0 | 保持「吃完更想拉」的特色循环 |
| 猫砂盆减少量 | -50 | 建议 -60 作为第一轮测试 | 让一次成功处理更有安全感 |
| 饱肚自然消耗率 | 0.25 / 秒 | 保留 0.25 / 秒 | 该压力节奏基本合理 |
| 场景交互时间 | 3 秒 | 保留 3 秒，先用地图距离调风险 | 交互时长可读，风险主要来自路径和敌人 |

这 7 个旋钮足够支撑 P0 第一轮调试。其他参数先不要频繁改，尤其不要同时改怪物伤害、怪物数量、屎意速率、饱肚速率和恢复量。否则测试结果会很难归因。

### 4.2 明确三条体验目标

P0 数值不应先追求「理论完整」，而应先追求三条体验指标。

| 体验目标 | 建议目标区间 | 观察方式 |
|-|-|-|
| 单层常规战斗时长 | 45 到 75 秒 | 超过 90 秒说明清怪和本能压力容易互相拖慢 |
| 猫砂盆使用频率 | 每 1.5 到 2 层处理 1 次 | 每层都必须处理会太碎，三层以上不处理会弱化特色 |
| 喂食器使用频率 | 每 2 到 3 层处理 1 次 | 太频繁会像杂务，太稀少则饱肚度存在感不足 |

这三条目标比单个数值更重要。只要玩家每 1 到 2 层会自然产生「要不要先去处理猫砂盆 / 喂食器」的判断，系统就成立；如果玩家每波都被迫处理，那节奏会被打碎。

### 4.3 给玩家一个早期安全阀

建议在第 1 到第 3 层加入明确的容错：第一次屎意满值不立刻降低睡眠度上限，而是触发「猫咪憋不住预警教学」，给玩家一次 10 秒引导去猫砂盆的机会。若玩家仍未处理，再执行正式拉屎惩罚。

这个安全阀不会削弱系统严肃性，因为它只发生在教学期一次。它的价值在于让玩家知道自己为什么受罚，而不是第一次见到屎意系统就被永久上限惩罚。

### 4.4 把地图距离纳入数值设计

猫砂盆和喂食器不是纯 UI 按钮，而是地图上的风险点。文档目前已经提出要确定坐标和与床的距离，这应提升为高优先级，因为 3 秒交互时间是否合理，取决于从床到猫砂盆需要跑多久、路上是否有怪、是否能被远程怪攻击。

建议用三个距离档位做测试：近距 2 秒路程、中距 4 秒路程、远距 6 秒路程。若猫砂盆在 6 秒外，20 秒倒计时会偏紧；若喂食器也在远处，玩家会倾向放弃补给。P0 最稳方案是猫砂盆略近、喂食器略远：屎意处理是硬需求，饱肚处理是规划需求。

## 5. 建议新增到详细数值文档的内容

### 5.1 增加「灰盒测试指标」章节

建议在详细数值文档后追加一节「测试指标与调参口径」，记录每轮测试必须采集的数据：

| 指标 | 记录目的 |
|-|-|
| 每层通关时长 | 判断怪物强度和清怪效率 |
| 每层睡眠度净变化 | 判断守床压力是否过重 |
| 每层猫砂盆使用次数 | 判断屎意压力是否过频 |
| 每层喂食器使用次数 | 判断饱肚压力是否有存在感 |
| 拉屎触发次数 | 判断惩罚是否太频繁 |
| 猫咪虚弱次数 | 判断血量和怪物攻击是否合理 |
| 玩家失败层数分布 | 判断难度曲线卡点 |

没有这些指标，后续调参会很容易变成主观争论。P0 阶段不需要复杂后台，只要在调试面板或日志里打印这些关键数值即可。

### 5.2 增加「P0 第一轮实现范围」

建议把完整设计拆成「第一轮必须实现」和「后续增强」。第一轮必须实现四项数值、床伤害、猫血量、猫砂盆、喂食器、基础 Debuff、简单虚弱。后续增强再加入连续虚弱惩罚、过量治疗转护盾、屎意倒计时延长、特殊梦境事件和复杂边界条件。

这样拆分能降低开发风险。数值系统最大的问题不是写不出规则，而是规则太多时很难知道哪个规则影响了手感。先做一版简单但能跑的循环，比一次性实现全规则更适合 P0。

## 6. 优先级建议

| 优先级 | 建议项 | 理由 |
|-|-|-|
| P0 必做 | 保留四核心数值，但降低早期屎意 / 睡眠上限惩罚 | 保住特色，同时避免新手早期崩盘 |
| P0 必做 | 建立 7 个核心调试旋钮 | 保证调参能归因 |
| P0 必做 | 明确猫砂盆和喂食器距离 | 场景交互风险必须和地图绑定 |
| P0 必做 | 简化虚弱规则 | 降低开发和 QA 成本 |
| P0 建议 | 增加灰盒测试指标 | 让调参从感觉变成证据 |
| P1 再做 | 连续虚弱叠加、过量治疗转护盾、复杂边界条件 | 完整但不是第一轮必要 |
| P1 再做 | 屎意和饱肚的连续曲线 Debuff | 手感更细腻，但 P0 分段足够验证 |

## 7. 最终建议

这套核心数值系统可以进入 P0，但不要按当前详细规则一次性全量实现。最稳的落地方式是：先保留四项数值和「吃→消化→拉」特色循环，先把惩罚做轻，把调试旋钮做少，把测试指标做明确。等守床、切猫、猫砂盆、喂食器四个基础行为都跑顺后，再逐步加重屎意、饱肚和虚弱惩罚。

如果要用一句话概括：当前设计的方向正确，问题不是不合理，而是太完整。P0 应该先做「可玩、可调、可归因」的轻量版，再把完整规则作为后续增强池。
